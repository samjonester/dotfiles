Which Layer for What

Five tools, each owning a specific concern. The key decision is which one manages your splits.

Kitty — OS-level: windows, tabs, GPU rendering, keyboard
tmux — Session persistence, pane layout, process isolation
Pi
AI agent + conversation
Neovim
Editor + LSP + lazygit float
lazygit
Git TUI (standalone)

The rule: Kitty tab = tmux session = activity. tmux owns everything inside.

Each Kitty tab maps 1:1 to a tmux session. Each session is an activity — coding a feature, triaging Slack, reviewing PRs, investigating an incident. You won't have all sessions open as tabs at once; tmux a -t name in any tab to bring one back.

Use thisFor thisWhy
kitty tab One per activity — coding a feature, triaging Slack, reviewing PRs, investigating an incident OS-level separation. Always renamed on creation (⌘ ⇧ I). Survives tmux kill-server.
tmux session The persistent activity context inside each Kitty tab. Stays alive through disconnects. Close a tab, tmux a -t triage in a new one to resume. Not all sessions need a tab at all times.
tmux windows Separate tasks within the activity — Pi in one, nvim in another, shell in a third Fullscreen per task. Switch with C-Space 1-9 or C-Space s.
tmux panes Side-by-side views within a task — Pi + a log tail, agent team layout, Pi + nvim Visible at the same time. Navigate with C-h/j/k/l. Zoom with C-Space z.
kitty splits Don't use inside tmux. They stack on top of tmux panes and create confusion. tmux owns all splits. Kitty splits only if you ever need a non-tmux tab for a quick one-off.
Typical layout: 2–3 Kitty tabs open — one for coding, one for triage, maybe one for investigation. Other tmux sessions stay detached; bring one back with tmux a -t name in a fresh tab. Every new tab gets renamed immediately (⌘ ⇧ I) — the tab name is the activity name.

Coding: The split-pane layout

Wide monitor, vertical splits. Pi on the left, something contextual on the right. Open the right pane via Pi commands (/nvim, /lazygit, /shell, /tmux-fork-pane) or manually: C-Space \. Close with C-Space x.

The left pane is always Pi. The right pane is whatever supports the current task — a second Pi (/tmux-fork-pane), neovim for browsing code and staging changes (⎵gg for lazygit float), standalone lazygit (/lazygit), nvim for browsing (/nvim), or a shell (/shell) — all auto-detect the right working directory. Servers run in a separate background window, or Pi spawns them as background processes. Close any right pane with C-Space x when done.

Tearing down sessions

Sessions accumulate. Here's how to clean up — from gentle to nuclear.

MethodHowWhat dies
Clean exit/exit in Pi → exit in shell. If single-pane window, session dies.That Pi process. Shell. Pane.
Kill one sessiontmux kill-session -t name (from another pane/terminal)All windows, panes, and processes in that session.
Kill all except onetmux kill-session -a -t mainEvery session except main.
Kill pane onlyC-Space x (confirm)Current pane and its process.
Kill windowC-Space & (confirm)All panes in the window.
Nucleartmux kill-serverEverything. All sessions, all processes. Kitty tabs survive.
What survives teardown: Memory bank (persists to disk), WTP worktrees (just directories — free with _wtp free <slot>), Kitty tabs and OS state. Session-scoped Pi cron tasks and background jobs die with their session — they're meant to be ephemeral.

Review & Planning

Multi-agent workflows that run inside Pi. Subagents are invisible background processes; agent teams spawn visible tmux panes.

Planning a feature

Say "plan this" — the multi-model pipeline runs automatically:

PhaseWhat happens

Reviewing code

Say "review this", "review PR #N", or "review current branch". For a full isolated environment: /pr 570491 sets up a WTP worktree + tmux window + dev up in one shot.

Forking & Parallel Work

Two kinds of fork depending on whether you need parallel execution.

In-place fork — branch the conversation pi

Ctrl+⇧ F or /fork — copies conversation history into a new session file. Same pane, same process. Good for "let me try a different approach" — go back with Ctrl+⇧ R.

tmux fork — parallel execution tmux pi

CommandResult
/tmux-fork [name]New tmux window with a forked Pi. Full conversation context. Runs in parallel.
/tmux-fork-pane [name]New tmux pane (side-by-side split). Full context. Both visible at once.
When to use which:
/fork — you want to explore a branch and might come back. No extra processes.
/tmux-fork — you want two agents working simultaneously on different things.
/tmux-fork-pane — same, but you want to see both at once (e.g., one investigates while the other monitors).

Troubleshooting

ProblemFix
Shift+Enter doesn't create newlinetmux show -g extended-keys → must be on. extended-keys-format → csi-u. Requires tmux kill-server.
Colors wrong in tmuxecho $TERM → xterm-256color. Check terminal-overrides has Tc.
Agent teams: "not in tmux"Pi must be inside a tmux pane. Start tmux first, then devx pi.
Pane titles not showingpane-border-status must be top. Only visible with 2+ panes.
C-h/j/k/l does nothingvim-tmux-navigator: passes through to vim if focused. Inside Pi editor, Pi handles them.
Images not renderingallow-passthrough on in tmux config.
Escape feels laggyescape-time 0 in tmux config.
/tmux-fork failsMust be in tmux with an active Pi session. /reload if extension not loaded.
Kitty splits inside tmuxDon't. Use tmux panes. Kitty splits only in non-tmux tabs.
